## ewizt Relay — Long-Poll Optimized

### LiteSpeed Connection Timeout Discovery
- ewizt.com runs LiteSpeed (not Apache/nginx)
- LiteSpeed has a hard **Connection Timeout of ~120s** — kills requests regardless of PHP `set_time_limit`
- `set_time_limit(0)` works in PHP, but LiteSpeed still cuts at 120s
- `sleep(120)` works fine (LiteSpeed doesn't timeout during sleep)
- Initial poll was 55s — conservative guess. Could push to 290s PHP, but LiteSpeed kills at 120s

### Final Settings
- **poll.php**: `set_time_limit(0)` + `$max_wait = 110` (safely under 120s LiteSpeed limit)
- **WSL relay-poller**: fetch timeout = 130s (110s + 20s buffer)
- Bandwidth: ~10.2 KB/hr idle (vs 20.5 KB/hr with 55s — 50% reduction)
- Poll cycles: ~33/hr (vs ~65/hr with 55s)

### PHP Debugging on ewizt.com
- `check.php` — test `max_execution_time`, `set_time_limit(0)`, `default_socket_timeout`
- `debug-poll.php` — standalone 110s loop test (confirmed LiteSpeed doesn't kill it mid-sleep)
- LiteSpeed kills PHP when the **connection itself times out** (not PHP execution)
- The 500 error at 120s was LiteSpeed's connection timeout, not PHP's fault

### Architecture Summary
- WSL initiates ALL connections outbound — no inbound ports needed, works through NAT/CGNAT
- Long-poll: WSL holds connection open for 110s, ewizt PHP holds until webhook or timeout
- Self-managing loop inside server process — no cron, no external scheduler
- Stale watchdog: restarts poller if no successful poll for 5min

### ewizt.com PHP Files (relay/)
- `shared.php` — SQLite helper, auth token `NsufkdwqJWfr4GwFko7R`

## Server-Managed Cron Scheduler Built & Tested

### New files
- `server/src/scheduler.ts` — Scheduler class (60s tick, auto-discover flows with `schedule:`)
- `server/src/flow-executor.ts` — shared `executeFlow()` / `executeFlowFrom()` extracted from router.ts

### How it works
- Runs inside server process as `setInterval` (every 60s)
- Auto-discovers flows with `schedule:` in YAML at startup via `bootstrap()`
- When cron time hits → triggers flow directly (no HTTP call) with `trigger_type: cron`
- Prevents duplicate runs (checks if successful run already happened for that cron slot)
- Auto-advances to next cron slot after execution
- Enable/disable individual schedules at runtime
- Env vars: `SCHEDULER_ENABLED`, `SCHEDULER_INTERVAL_MS` (default 60000), `SCHEDULER_TIMEZONE`

### API endpoints added to router.ts
- `GET /schedules` — list all scheduled flows (cron, next run, last run, enabled)
- `GET /schedules/:name` — detail + recent run history
- `POST /schedules/:name/enable` — re-enable disabled schedule
- `POST /schedules/:name/disable` — pause schedule
- `POST /schedules/:name/run` — manually trigger now

### Tested
- Flow with `cron: "* * * * *"` triggered exactly on the minute ✅
- Run logged with `trigger_type: cron` ✅
- Enable/disable works ✅
- Manual trigger works ✅
- Schedule advances to next slot automatically ✅

### Watchdog note
- Watchdog still monitors/alerts but does NOT execute flows (scheduler does that now)
- `expected_runs` table shared between watchdog and scheduler
- Old `test-schedule` flow was deleted but left a stale `expected_runs` entry — harmless
