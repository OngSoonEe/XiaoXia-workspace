## Critical Discovery: OpenClaw vs Lobster Bot Token Conflict

**Root cause of all Telegram command delivery failures:** OpenClaw and the lobster webhook handler both use the same bot token (`8399943065:AAGD4XmsRpmxJLDnZxQU-QR6TmwVb_nUKdI` — `jackoc1_bot`).

- OpenClaw runs in **long-polling mode** which requires `webhook.url = ""`
- Every `setWebhook` call to point the bot at the ewizt relay gets **immediately overwritten** by OpenClaw's next polling cycle
- This is why Jack's `/task-today a` commands in the group never reached lobster — the webhook was always empty
- The `setMyCommands` persistence issue (commands returning `ok:true` but `getMyCommands` returning `[]`) is likely the same conflict — OpenClaw is also managing commands

### Evidence
- `getWebhookInfo` returns `url: ""` even immediately after `setWebhook` returns `ok:true`
- No `deleteWebhook` calls in any lobster/MyTasks code — OpenClaw itself is clearing it
- OpenClaw's Telegram polling loop is the culprit

## Lobster Handler Bug Fixes (from earlier in session)

Three bugs were fixed in `telegram-command-handler.ts`:
1. **Regex hyphen bug**: `\w+` doesn't match `-`, so `/task-today` parsed as command `task` + args `-today`. Fixed to `[\w-]+`
2. **Chat ID type mismatch**: `ALLOWED_CHAT_IDS` had strings, Telegram sends numbers. `Array.includes()` does strict comparison. Fixed to use `Number(chatId)` and numeric array
3. **`reply` variable scope**: `let reply: string` was only declared inside `else` branch. Hoisted before `if/else` block
4. **`sendToChatId` logging bug**: `sendMessage(sendToChatId, ...)` was correct but log line printed `replyChatId` instead of `sendToChatId` — fixed the log
