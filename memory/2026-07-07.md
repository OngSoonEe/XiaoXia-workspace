## Lobster Automation — EADDRINUSE Fix & Network Issues (13:00–14:18)

### EADDRINUSE Root Cause & Fix — ✅ COMMITTED
- **Root cause:** When server crashes (SIGKILL/OOM), kernel keeps TCP socket in `TIME_WAIT` for 60s. systemd's `RestartSec=5s` fires before TIME_WAIT clears → `EADDRINUSE` → 5 rapid restart failures → systemd `StartLimitBurst=5` gives up → server stays dead for hours.
- **Fix applied (committed `a8b1f2c`):**
  - `server/src/server.ts`: Added `reuseAddress: true` to fastify listen options (enables `SO_REUSEADDR`)
  - `~/.config/systemd/user/lobster-server.service`: Changed `RestartSec=5s` → `RestartSec=65s` (wait for TIME_WAIT to clear), added `StartLimitBurst=3` + `StartLimitIntervalSec=300`
  - `KillMode=mixed` → `KillMode=control-group` (ensures child processes killed with parent)
- **Tested:** SIGKILL'd server process → systemd waited 65s → restarted successfully. Fix works.

### Flow Execution Bugs Found & Fixed
- **Bug:** `upload-status-page.ts` hangs forever when FTP to ewizt.com times out — no timeout on Python ftplib call. Server's `exec()` also has no timeout on step scripts.
- **Fix:** Added 20s socket timeout to Python FTP script; added `timeout: 120_000` (2min) to step-runner exec calls in `server/src/step-runner.ts`.
- **Result:** Flow now completes in ~26s instead of hanging 69s then failing.

## Lobster Automation — Relay Poller Backoff Fix & ewizt.com Block Resolution (21:00–23:31)

### Root Cause of ewizt.com IP Block — IDENTIFIED & FIXED
- **Root cause:** Relay poller's aggressive retry loop with no backoff. When ewizt.com unreachable, poller retried every 3s = ~4-5 req/min, 24/7. Generated **1,998 failed poll requests** + **480 stale restarts** over ~7 hours. This triggered firewall/IDS IP block at TCP level (all ports: 21, 80, 443).
- **Fix (committed `21dc55f`):** Added exponential backoff to relay poller in `server/src/server.ts`:
  - Backoff sequence: 3s → 6s → 12s → 24s → 48s → 96s → 192s → 300s (cap)
  - Resets to 3s on successful poll
  - Stale detector interval increased from 5s check to 60s (only checks timestamp, no outbound)
  - Max failure rate: ~12 req/hour (down from ~300 req/hour)
