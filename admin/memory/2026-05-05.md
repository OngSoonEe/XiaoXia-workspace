# JGS Quantization Re-Analysis — 2026-05-05 (17:26 SGT)

## Task
Jack requested thorough AI specialist re-analysis of three documents (UR, QR, cross-check) for Intel JaguarShore quantization, with research references. Email results to jack.ong.curio@gmail.com.

## Key Findings

### UR Context (Critical Correction from Prior Analysis)
- The UR contains **8 FINETUNE use-cases** (not simple inference recipes): Dense 70B SFT+PO, Flux image gen, RAFT, VLM SFT, MoE expert expansion, Agentic RL, distillation, code+safety alignment
- Prior analysis incorrectly treated these as simple chatbot inference recipes with models like Whisper-large, Qwen2.5-VL-3B, SD-3.5, Phi-3.5-mini — those were from a DIFFERENT UR document/image
- The correct UR models: Llama-3.1-70B, Flux.1-schnell, Llama4 Maverick, DeepSeek-V3, DeepSeek-Coder-V2

### Hardware Architecture
- **JGS 8-bit = MXFP8** (same as CRI)
- **JGS 4-bit = NVFP4** (NOT MXFP4; MXFP4 is CRI-only)
- QR v0.8.2 (Dec 2025) explicitly re-prioritized NVFP4 for JGS
- JGS supports native FP8 (CRI does not)

### Gap Summary
| Priority | Gap | Status |
|----------|-----|--------|
| P0 BLOCKING | DeepSeek-V3 (671B MoE) has ZERO quantization recipes in QR | No MXFP8 or NVFP4 recipe exists |
| P1 HIGH | DeepSeek-Coder-V2 (~236B MoE) has no recipe | NVFP4 needed to fit all experts in 160GB |
| P1 MEDIUM | Llama4 Maverick NVFP4 is WIP | Native FP8 model works today |
| P2 LOW | Flux NVFP4 not started | Not needed for Schnell (12B, already distilled) |

### AutoRound Research
- AutoRound (SignRound V1/V2) is the primary quantization algo for Intel GPUs
- NVFP4 g16 AutoRound accuracy validated (Llama3.1-8B: 0.6918 avg, close to MXFP4 g32 AutoRound at 0.6686)
- AutoRound supports diffusion quantization (transformer-only, Flux) but loadable format not ready yet
- AutoRound VLM quantization: only 5 model families supported, Llama4 Maverick not included
- Intel Extension for PyTorch is being retired — should not recommend

### UR Issues Found
- "TorchAO (quant optional)" is dangerously vague — doesn't specify JGS-native formats
- No FP8 KV Cache / FP8 Attention quantization specified for any LLM use-case
- DeepSeek-V3 vs DeepSeek-R1 confusion (different architectures, only R1 has recipes)
- No distinction between inference quantization vs training quantization
- No MoE loading strategy specified (active-only vs all-experts)
